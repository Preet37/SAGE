# Card: FlashMask mask representation + kernel integration
**Source:** https://arxiv.org/html/2410.01359v1  
**Role:** paper | **Need:** COMPARISON_DATA  
**Anchor:** Procedure/architecture for supporting richer masking patterns in FlashAttention-style fused kernels; mask representation + overheads

## Key Content
- **Masked attention math (Eq. 1/2):** scaled dot-product attention uses logits  
  \[
  S = \frac{QK^\top}{\sqrt{d_k}} + M,\quad P=\text{softmax}(S),\quad O=PV
  \]
  where \(Q,K,V\) are sequences, \(d_k\) head dim, \(M\in\mathbb{R}^{L\times L}\) mask with masked entries set to \(-\infty\) so softmax outputs 0 at those positions (Section 4.4).
- **Core observation (Section 3, Eq. 3):** for many practical masks, **masked rows per key-column are continuous intervals**, enabling 2D dense mask \(\to\) 1D interval representation: for column \(j\), masked row interval(s) \(I_j\subseteq[0,L)\).
- **FlashMask representation (Section 4.1):** partition attention matrix into lower-left vs upper-right triangles; represent mask with **four 1D vectors** (length \(L\)):  
  - \(l_s, l_e\): lower-triangular masked row start/end  
  - \(u_s, u_e\): upper-triangular masked row start/end  
  Each column has up to **two masked intervals**: \([l_s[j], l_e[j]]\) and \([u_s[j], u_e[j]]\).
- **Kernel procedure (Section 4.2):**
  1) **Preprocess in HBM:** block columns; compute per-block min/max of each vector \(\Rightarrow\) **8 vectors** \((l_s^{min/max}, l_e^{min/max}, u_s^{min/max}, u_e^{min/max})\), each size \(\approx L/B_c\).  
  2) **Real-time block classification (Eq. 4):** during tiled attention, classify each tile as **fully masked / unmasked / partially masked** using these min/max bounds; **skip fully masked tiles**, apply elementwise masking only for partial tiles.
- **Complexities (Section 4.3):**
  - Space: dense mask \(O(L^2)\) vs FlashMask \(O(L)\) (4 vectors + 8 min/max vectors).  
  - Compute: standard \(O(L^2)\) vs \(O(L^2(1-s))\) by skipping \(s\) fraction of fully masked blocks (block sparsity \(s\)).  
  - Memory access: dense mask requires \(O(L^2)\) HBM accesses; FlashMask reads mask vectors \(O(L)\) and can preload into SRAM.
- **Correctness (Section 4.4/5.2):** **bit-level numerical equivalence** to dense-mask FlashAttention when deterministic control enabled; skipping fully masked blocks doesn’t change outputs.
- **Empirical results (A100 80G, BF16):**
  - End-to-end throughput vs FlashAttention dense mask: **1.65×–3.22× speedup** (Section 5.1).  
  - Kernel TFLOPs/s vs FlexAttention: **+12.1% to +60.7%**, reaching **37.8%–62.3%** of A100 theoretical peak (Abstract/Section 5.4).  
  - Memory: at **64K**, dense mask overhead **8GB**; FlashMask enables **up to 544K** seq length in Llama-2 7B LoRA where others limited to 64K (Section 5.1).

## When to surface
Use when students ask how padding/causal/custom masks are implemented inside fused FlashAttention kernels, how to avoid dense \(L\times L\) masks, or what overhead/speedups sparse mask representations achieve.
