A Benchmark for Evaluating Repository-Level Code Agents with Intermediate Reasoning on Feature Addition Task
Abstract.
Repository-level code agents have shown strong promise in real-world feature addition tasks, making reliable evaluation of their capabilities increasingly important. However, existing benchmarks primarily evaluate these agents as black boxes based on final test correctness, providing limited insight into how they reason and where failures arise. To address this limitation, we introduce RACE-bench, a reasoning-augmented benchmark for evaluating code agents on repository-level feature addition tasks. RACE-bench contains 528 real-world feature addition instances from 12 open-source repositories. Each instance is paired with executable patch verification and structured intermediate reasoning ground truth covering issue understanding, file localization, implementation tasks, and step decomposition. Based on this design, we introduce a dual-track evaluation framework that jointly measures patch correctness and intermediate reasoning quality. We evaluate three representative repository-level code agents on RACE-bench. On the full benchmark, Resolved Rate ranges from 29% to 70% across different agents. Our reasoning-level analysis further shows that while current agents perform well at understanding high-level intent, their performance degrades substantially when translating intent into concrete implementation steps. We also find that apply-success but test-fail cases exhibit lower reasoning recall (35.7% decrease) and higher over-prediction (94.1% increase) compared to successful cases. These findings highlight the importance of evaluating repository-level code agents beyond final patch correctness by examining the quality of their reasoning processes.
1. Introduction
Recent years have witnessed the rapid growth of large language models (LLMs), giving rise to powerful models such as ChatGPT (Achiam et al., [2023](#bib.bib1)) and Gemini (Team et al., [2023](#bib.bib2)). These models have quickly reshaped automated software engineering and are now widely applied to code generation and related development tasks (Guo et al., [2024](#bib.bib3); Roziere et al., [2023](#bib.bib4); Lin et al., [2024](#bib.bib5); Yang et al., [2024c](#bib.bib6)). This technological shift has influenced real-world development practices: a substantial proportion of developers regularly rely on LLM-based tools (e.g., GitHub Copilot (GitHub, [2021](#bib.bib10))) to write, modify, or review code (Wermelinger, [2023](#bib.bib7); Barke et al., [2023](#bib.bib8); Dakhel et al., [2023](#bib.bib9)). These tools significantly improve productivity and reduce cognitive load during programming activities (Joshi et al., [2023](#bib.bib11); Vaithilingam et al., [2022](#bib.bib12)).
Beyond single-turn code generation, recent research has introduced code agents that extend LLMs into autonomous, multi-step problem-solving systems capable of interacting with real-world code repositories (Gao et al., [2025](#bib.bib18); Zhang et al., [2024](#bib.bib19); Wang et al., [2024](#bib.bib20)). Unlike LLMs that generate code in isolation, code agents iteratively reason about a given problem, invoke external tools (e.g., code search, testing frameworks, and execution environments), and adapt their behavior based on intermediate feedback. As a result, they can perform a broad range of repository-level development tasks, including fixing bugs and implementing feature requests (Bouzenia et al., [2024](#bib.bib21); Zhang et al., [2024](#bib.bib19); Wang et al., [2024](#bib.bib20); Yang et al., [2024b](#bib.bib22)), generating tests from natural-language issue descriptions (Mündler et al., [2024](#bib.bib23); Ahmed et al., [2024](#bib.bib24); Nashid et al., [2025](#bib.bib25)), and automating complex project setups (Hu et al., [2025](#bib.bib26); Bouzenia and Pradel, [2025](#bib.bib27)).
Among these tasks, feature addition is particularly important in real-world development, as it requires understanding natural-language feature requests and implementing coordinated modifications across multiple files and code entities, thereby posing a comprehensive challenge for code agents. This trend highlights the need to systematically evaluate the capabilities of code agents in real-world feature addition scenarios.
However, the failure modes of code agents are more complex compared with traditional code-generation LLMs. In real-world feature addition workflows, producing a correct patch is the outcome of a structured reasoning pipeline that involves ❶ understanding the issue and its requirements, ❷ identifying the relevant files, ❸ locating the necessary code entities, and ❹ formulating an appropriate modification plan. Errors can arise at any stage of this process: an agent may misunderstand the problem intent, modify irrelevant files, introduce unnecessary changes, or omit essential steps. Moreover, code agents may occasionally pass test cases despite relying on coincidental reasoning, leading to apparent success that hides underlying problems. As a result, evaluating code agents solely based on whether the final patch passes tests, without examining the intermediate reasoning process, offers limited insight into their true capabilities and failure characteristics.
Existing benchmarks for evaluating code LLMs and code agents fall short in capturing the intermediate reasoning challenges in repository-level feature addition scenarios. Function-level benchmarks such as HumanEval (Chen, [2021](#bib.bib13)), MBPP (Austin et al., [2021](#bib.bib14)), and LiveCodeBench (Jain et al., [2024](#bib.bib15)) focus on small, context-independent function generation tasks. While effective for evaluating isolated code generation capabilities, these benchmarks fail to reflect real-world development, where changes often span across multiple interdependent files and code entities. To bridge this gap, recent benchmarks such as SWE-bench (Jimenez et al., [2023](#bib.bib16)) and FEA-bench (Li et al., [2025](#bib.bib17)) derive tasks from real issues and patches in open-source repositories. However, although these benchmarks represent a step toward more realistic evaluation, they focus solely on whether the final generated patch passes the tests. This black-box evaluation paradigm offers limited visibility into an agent’s intermediate reasoning process and makes it difficult to diagnose where and why an agent fails within the development workflow.
To address this gap, we present a new benchmark named RACE-bench (Reasoning-Augmented Code Agent Evaluation) for systematically evaluating code agents in realistic repository-level feature addition scenarios with explicit reasoning evaluation. RACE-bench is designed with the following key characteristics: ➀ RACE-bench is constructed from high-quality, actively maintained open-source repositories, with each instance grounded in a real feature request and its corresponding pull request (PR). ➁ RACE-bench focuses on feature addition tasks, which require agents to infer intended functionality from natural-language issues and coordinate changes across multiple files and code entities. ➂ RACE-bench provides structured ground truth for intermediate reasoning stages, including issue understanding, relevant file identification, concrete implementation tasks, and abstract task-step decomposition. ➃ Each task is equipped with the Fail-to-Pass (FTP) and Pass-to-Pass (PTP) test cases, verifying both the correctness of generated code and the preservation of original functionality. ➄ RACE-bench enables fine-grained evaluation beyond final patch correctness by measuring reasoning recall and over-prediction (e.g., redundant reasoning steps) at different modules of the development workflow. ➅ RACE-bench provides a unified, containerized execution environment for each instance using Docker, enabling reliable reproduction of repository states and consistent evaluation across different agents. Overall, RACE-bench contains 528 feature addition instances derived from 12 actively maintained open-source GitHub repositories. To facilitate more efficient experimentation, we construct a lightweight subset named RACE-bench Lite. This subset contains 100 instances selected through a difficulty-aware sampling strategy designed to preserve the overall challenge level of the RACE-bench.
To evaluate the effectiveness of RACE-bench, we assess three representative code agents on the full benchmark using DeepSeek as the base model. The results reveal substantial performance variation across agents, with Resolved Rate ranging from 29% to 70%. Beyond final patch correctness, our reasoning evaluation shows that while code agents exhibit strong high-level intent comprehension, their performance degrades in a waterfall-like manner when translating goals into concrete tasks and execution plans. Furthermore, apply-success but test-fail cases are characterized by lower recall (a 35.7% decrease) and higher over-prediction (a 94.1% increase) compared with successful cases. We further extend the evaluation on RACE-bench Lite by incorporating additional base LLMs. The results indicate that although the choice of base LLM influences agent performance, its impact is constrained by the agent architecture. Agents with better-designed frameworks maintain relatively stable performance across different base LLMs.
Contributions: In summary, the main contributions of this paper can be summarized as follows:
-
•
We introduce RACE-bench, a benchmark for evaluating repository-level code agents on feature addition tasks. It contains 528 real-world feature addition instances from 12 GitHub repositories, each equipped with executable patch verification and structured intermediate Reasoning Ground Truth.
-

2. RACE-bench
In this section, we present the design and construction of RACE-bench. We first provide an overview of the benchmark and its instance structure, then detail the benchmark construction pipeline. We subsequently describe the evaluation setting and conclude by introducing the evaluation metrics.
2.1. Overview
The task instances in RACE-bench are constructed from real-world PR data and their corresponding GitHub issues. As illustrated in Figure [1](#S2.F1), an instance supports the evaluation of both final patch correctness and intermediate reasoning quality. Each instance consists of three primary components:
❶ Task Context. This module provides the necessary information and execution environment to initialize a development task. The Feature Request is derived from the original GitHub issue and provides a natural language description of the intended functionality. The Environment Setup specifies the execution context, including the repository, release version, and base commit. This setup ensures that evaluations are conducted in a consistent and reproducible state. Additionally, the module may include optional Hints Text from developer discussions to provide extra context for design decisions.
❷ Reasoning Ground Truth. To evaluate the reasoning quality of code agents, RACE-bench follows a human-like workflow that decomposes the reasoning process into four stages containing five reasoning modules:
-
•
Issue Understanding. This component captures the understanding of the issue intent. The Concept Explanations identify domain-specific concepts, while the Goal Expectation provides a concise summary of the expected behavior.
-
•
File Localization. This component identifies the files that are modified or added. All files are categorized into three types. Necessary Code Files are essential for the patch to pass the tests. Other Files include auxiliary changes like documentation or changelogs. Test Files consist of the unit test files for the task instance.
-
•
Issue Implementation. This component identifies the code entities to be added, modified, or removed. For each entity, it includes the file location and the change type (i.e., new, modified, or deleted), along with its purpose and an is_necessary flag to indicate whether it is an indispensable entity to pass the tests.
-
•
Step Decomposition. This component abstracts the implementation process into a sequence of essential task steps. Each step describes a high-level modification action (e.g., introduce new capability and reuse existing semantics).
❸ Verification. This module evaluates the functional correctness of a generated patch through automated testing. Each instance provides a Gold Patch and an associated Test Patch for the requested feature from the PR. These patches can be applied to the repository using the git apply tool. The correctness is verified by executing the Unit Test with pytest. The FTP test cases fail before applying the Gold Patch and pass afterward. The PTP test cases ensure that existing functionalities remain preserved after the modification.
2.2. Benchmark Construction
This subsection describes the end-to-end construction pipeline of RACE-bench, including repository selection and environment setup, instance filtering, Reasoning Ground Truth construction, and the validation process to ensure data quality and reliability.
2.2.1. Repository Collection and Environment Construction
Following the design of SWE-bench (Jimenez et al., [2023](#bib.bib16)), we select the same set of 12 popular open-source Python repositories on GitHub as the source for RACE-bench instances. These repositories are widely used in practice, actively maintained, and governed by well-defined contribution guidelines.
To ensure that both instance construction and subsequent benchmark evaluation are performed in a stable and reproducible environment, we build independent Environment Docker images for each stable release of the repositories. Each image encapsulates the required dependencies and Python runtime. This approach mitigates environment drift and ensures that tests are executed consistently across experiments.
2.2.2. Instance Filtering
Starting from all merged PRs across the selected 12 repositories, we apply a three-stage filtering pipeline to identify high-quality feature addition instances suitable for evaluating repository-level code agents.
Attribute-based filtering. We first perform coarse-grained filtering based on observable PR attributes. Specifically, we retain only PRs that (1) modify files under test-related directories (e.g., test or testing), indicating the presence of explicit test changes, and (2) are explicitly linked to at least one GitHub issue, ensuring the availability of a natural-language feature request. After this stage, 14,333 PRs satisfy the filtering criteria.
Intent-based filtering. Next, we identify PRs that correspond to feature addition tasks. For each remaining PR, we use DeepSeek to classify the development intent based on the PR title and description, assigning it to categories such as new feature, bug fix, or refactor. To assess the reliability of the classifier, we randomly sample 100 PRs from the filtered pool and manually verify their predicted labels. The manual inspection shows that the classification accuracy exceeds 99%, indicating that the LLM provides reliable intent predictions for this task. We retain only PRs whose intent is classified as new feature and manually recheck the candidate PRs to confirm that they correspond to feature addition tasks. This filtering step yields 2,931 candidate PRs.
Execution-based filtering. Finally, we apply an execution-driven validation process to ensure that each candidate PR can serve as a reliable benchmark instance. For each PR, we retrieve its corresponding base commit and repository version, and perform the following steps within the matching Environment Docker: (1) clone and install the repository; (2) check out the base commit; (3) apply the Test Patch and execute the test suite, where at least one test is expected to fail; (4) apply the Gold Patch and re-run the tests; (5) compare the test outcomes before and after applying the Gold Patch to determine the FTP and PTP test sets. If any step fails (e.g., environment setup errors or missing FTP tests), the instance is discarded. After this stage, 580 PRs remain and are retained as valid RACE-bench instances.
2.2.3. Reasoning Ground Truth Construction
For each retained instance, we construct a structured Reasoning Ground Truth to enable evaluation of intermediate reasoning behaviors exhibited by code agents.
The ground truth is organized according to the four-stage reasoning schema described in Section [2.1](#S2.SS1), and is derived through a combination of LLM assistance and patch-driven program analysis.
Issue Understanding. The Concept Explanations and Goal Expectation are generated using the DeepSeek LLM. We provide the model with the original Feature Request, the Hints Text, and the repository README to supply task-specific and project-level context. The model identifies domain-specific concepts from the issue description and summarizes the expected system behavior after implementation. These summaries focus on observable behavior. To prevent implementation leakage, we explicitly prohibit code-level details.
File Localization. The File Localization ground truth is extracted from the Gold Patch and Test Patch. All files modified or added in the Gold Patch are initially marked as relevant. To distinguish necessary files from others, we employ a test-driven ablation procedure. For each modified file, we create a patch variant by removing the changes of that file and then re-executing the Unit Test. A file is labeled as a Necessary Code File if removing its changes causes a test failure; otherwise, it is labeled as an Other File. In addition, all files modified or added in the Test Patch are labeled as Test File.
Issue Implementation. The Specific Tasks ground truth captures the set of code entities required to implement the feature. For each instance, we analyze the Gold Patch by statically parsing the modified source files to extract added, modified, or deleted code entities (i.e., functions and methods). We define classes as a structural container rather than a code entity, as they typically consist of multiple methods. For each extracted entity, we further annotate its semantic purpose using DeepSeek. Specifically, we provide the model with the original Feature Request and the patch-local code context of the entity, and instruct the LLM to summarize the role of the entity in terms of observable behavior. To determine whether an entity is semantically necessary, we additionally provide the FTP tests and ask the model to assess whether the intended behavior can still be achieved without modifying that entity.
Step Decomposition. Finally, we construct the Task Steps ground truth by abstracting the gold implementation into a minimal sequence of essential semantic steps. We define a fixed and closed taxonomy of step types (i.e., introduce new capability, reuse existing semantics, change existing semantics, deprecate or replace behavior and enforce constraints or edge cases). This taxonomy captures common categories of externally observable implementation changes required for tests to pass. Given the filtered Specific Tasks, the Gold Patch, and the original Feature Request, DeepSeek is prompted to select and order a minimal set of task steps from this taxonomy. Each step represents exactly one irreducible semantic responsibility, is grounded in one or more existing tasks, and is assigned a single step type. This representation abstracts low-level code details while preserving the essential structure and ordering of the implementation process.
2.2.4. Validation
To ensure the reliability of the automatically constructed Reasoning Ground Truth, we perform an LLM-assisted audit over 580 candidate instances that remain after instance filtering, followed by targeted human inspection on a subset of cases.
For each candidate instance, a GPT 5.2 LLM auditor evaluates three aspects: (1) For Issue Understanding, we assess whether the Concept Explanations accurately cover the domain-specific terms appearing in the Feature Request and whether the Goal Expectation correctly summarizes the intended requirement. (2) For Specific Tasks, we verify whether entities labeled as necessary correspond to genuine requirements of the Feature Request and whether their inferred purposes are semantically accurate. (3) For Step Decomposition, we assess whether each Task Step represents a distinct and essential semantic step by performing an ablation check. In this process, a step is considered valid only if its removal renders the Feature Request unsatisfiable.
Instances are removed if the auditor detects semantic misinterpretation, incorrect necessity judgments, or structural defects. To assess the reliability of this automatic audit, we additionally conduct targeted human validation on 100 audited instances, including 50 instances accepted by the auditor and 50 instances rejected by the auditor, sampled across different repositories. Two authors independently inspect these cases and then resolve disagreements through discussion. On this subset, the human inspectors agree with the auditor’s decision on 96 out of 100 cases (96.0%), suggesting that the automatic audit is generally reliable for filtering problematic Reasoning Ground Truth. In total, 52 out of the 580 candidate instances are excluded. After this stage, 528 validated instances remain and form the full RACE-bench.

2.2.5. RACE-bench Lite Construction
Evaluating repository-level code agents requires substantial LLM interaction, making large-scale evaluation costly. To enable efficient experimentation while preserving the overall difficulty distribution of the benchmark, we construct a lightweight subset, RACE-bench Lite.
Starting from the validated instances in the full benchmark, we select a subset of size (default ) whose empirical difficulty closely matches that of the full benchmark. Specifically, for each instance we collect two results from the full benchmark runs of three agent configurations (AutoCodeRover, TraeAgent, and mini-SWE-Agent): (1) whether the generated patch passes the test suite (patch success) and (2) whether the patch can be applied successfully (apply success). 2.3. Evaluation Workflow
We propose a dual-track evaluation workflow to systematically assess both the final patch output and the intermediate reasoning process of code agents. As illustrated in Figure [2](#S2.F2), the workflow consists of three stages: agent inputs, code agent execution, and dual-track evaluation.
Agent Inputs.
A code agent is initialized in the repository state defined by the Environment Setup in Figure [1](#S2.F1).
The agent is provided with the Feature Request and a set of FTP tests as observable inputs.
These FTP tests serve as an executable functional specification, defining the minimal correctness criteria that the agent’s implementation must satisfy.
Code Agent Execution. A code agent typically operates through the coordination of an agent controller, external development tools, and an underlying LLM. Commonly used tools include file search, static analysis, and test execution. Guided by the LLM’s intermediate outputs, the agent iteratively invokes these tools to inspect the codebase, validate hypotheses, and gather task-relevant information. This reasoning–action loop continues until the agent generates a final patch. The interaction traces between the agent controller and the LLM record the decisions and rationales underlying the code changes, and serve as the primary evidence for analyzing the agent’s reasoning behavior.
Dual-Track Evaluation. The evaluation of an instance consists of two complementary components:
-
•
Patch Verification: The agent is expected to generate a patch that satisfies all FTP tests while preserving existing behaviors validated by PTP tests. A patch is considered functionally correct if it passes all required tests in the evaluation environment.
-
•
Reasoning Assessment: We employ a summarizer LLM to post-process the interaction traces between the agent controller and the underlying LLM. These traces are converted into a structured representation that aligns with the predefined Reasoning Ground Truth schema. If the agent successfully produces a valid patch, we further enrich the summarized reasoning by applying the same patch-driven back-inference procedure used to construct the Reasoning Ground Truth, extracting additional information such as file localization, specific implementation tasks, and abstract task steps. Finally, the resulting structured reasoning is systematically compared with the Reasoning Ground Truth. Detailed evaluation metrics are presented in Section
[2.4](#S2.SS4).
2.4. Evaluation Metrics
We evaluate agents on the proposed benchmark from two complementary perspectives: Patch Evaluation and Reasoning Evaluation. Patch Evaluation focuses on the correctness of the generated code, while Reasoning Evaluation assesses the quality of the intermediate reasoning produced during the problem-solving process.
Patch Evaluation.
An agent is considered successful on an instance if the generated patch passes all FTP and PTP tests. Following prior work on repository-level code generation benchmarks (Li et al., [2025](#bib.bib17); Chen et al., [2025](#bib.bib28); Jimenez et al., [2023](#bib.bib16); Deng et al., [2025](#bib.bib29)), we adopt Resolved Rate (%) as the primary metric for evaluating patch correctness. This metric measures the percentage of instances for which the agent produces a successful patch on the first attempt.
We also report the Patch Apply Rate (%), which captures the proportion of generated patches that are syntactically valid and can be applied to the repository without errors.
Reasoning Evaluation. Table [1](#S2.T1) presents the evaluation metrics used for each reasoning module. For a subset of the intermediate reasoning modules that require semantic judgment, we employ an LLM-based evaluation paradigm (LLM-as-Judge). The reliability of LLM-as-Judge has been widely studied and shown to be effective not only in natural language generation tasks (Liu et al., [2023b](#bib.bib30); Song et al., [2024](#bib.bib31)), but also in software engineering-related tasks (He et al., [2025](#bib.bib32); Wang et al., [2025](#bib.bib33)).
To compute the metrics for Specific Tasks, it is necessary to align benchmark tasks with the tasks inferred from the agent-generated patch. We adopt a hybrid alignment strategy that combines deterministic matching with semantic matching.
For tasks whose Change Type is deleted or modified, we apply deterministic alignment: a task is considered aligned if both its Entity Name and Location match exactly between the benchmark and the agent output.
For tasks with Change Type new, we first attempt deterministic matching using the same criteria. If no aligned task is found, we further employ an LLM-based semantic matcher to compare the Purpose descriptions and identify an aligned task at the functional level.
A similar alignment procedure is applied to the Task Steps module. For each benchmark step, we first select candidate agent steps that share the same Step Type and have overlapping Related Tasks. We then use an LLM-based semantic judgment to determine whether any candidate step covers the benchmark step’s intent.
3. Experimental Setup
3.1. Agents and Models
Agent selection.
We evaluate three representative repository-level code agents selected from the SWE-bench leaderboard (SWE-bench, [2024](#bib.bib48)), which provides a widely used reference point for current automated software engineering agents.
Specifically, we include AutoCodeRover (Zhang et al., [2024](#bib.bib19)), a research system that emphasizes structured repository navigation and localized edits; TraeAgent (Gao et al., [2025](#bib.bib18)), which investigates test-time scaling and iterative refinement for software engineering tasks; and mini-SWE-Agent (Yang et al., [2024b](#bib.bib22)), a lightweight agent framework derived from SWE-Agent that integrates LLM reasoning with repository execution environments.
We select these agents because they are widely used in SWE-bench evaluations, represent both research-oriented and practical agent designs, and have publicly documented workflows that enable reproducible evaluation.
Model selection.
We use DeepSeek V3.2 (Liu et al., [2024a](#bib.bib49)) as the base model and evaluate it across all three agents on the full RACE-bench.
To broaden model coverage under a lower-cost setting, we additionally report the results of three agents on RACE-bench Lite using GLM-4.7 (Zhipu AI, [2025](#bib.bib50)) and GPT-5.2 (OpenAI, [2025](#bib.bib51)) as base models.
In addition, DeepSeek V3.2 is used as the summarizer LLM to generate a normalized intermediate reasoning chain for the reasoning evaluation.
3.2. Setup
Execution environment.
To ensure reproducibility, we evaluate each instance in an isolated Instance Docker built on the Environment Docker image of the related release version described in Section [2.2.1](#S2.SS2.SSS1).
For every instance, we reset the repository to its base commit, clone the target project, and perform the required project setup steps (e.g., installing the package with pip install -e .).
Hardware. All experiments are conducted on a Linux server equipped with two Intel Xeon Platinum 8358P CPUs (64 cores total @ 2.60 GHz) and 2 TB RAM, running Ubuntu 20.04 LTS.
Agent configuration. For all agents, we use their default configurations, following the recommendations in the official documentation. To compute Resolved Rate, we run each agent on each instance exactly once (i.e., single attempt per instance).
LLM configuration. All LLM calls in this work are made through official APIs. We set temperature=0 and top_p=1. The maximum generation length is set to 4,096 tokens for agent inference and 8,192 tokens when the model is used as the summarizer LLM for intermediate reasoning normalization.
4. Results and Analysis
We organize our evaluation around three research questions (RQs):
RQ1. (Capability) How well do code agents resolve repository-level feature addition tasks on RACE-bench?
RQ2. (Reasoning) How well do code agents perform on intermediate reasoning evaluation, and what does the reasoning profile reveal about their strengths and limitations?
RQ3. (Generality) How does the choice of the agent’s base LLM affect performance on RACE-bench Lite?
4.1. RQ1: Capability
RQ1 evaluates the patch-level capability of code agents to resolve repository-level feature addition tasks.
We evaluate three agents on the full RACE-bench and measure their end-to-end effectiveness by whether the generated patch can (i) be applied cleanly and (ii) pass the benchmark tests.
Table [2](#S4.T2) summarizes the results.
Main results. AutoCodeRover resolves 28.79% (152/528) of the instances while achieving a very high Patch Apply Rate of 96.21% (508/528). This suggests that AutoCodeRover is generally reliable at producing syntactically valid patches, but it often fails to fully satisfy the functional requirements enforced by the test suite. TraeAgent attains a notably higher Resolved Rate of 52.65% (278/528), indicating stronger task-solving effectiveness. However, its Patch Apply Rate is substantially lower at 78.98% (417/528). mini-SWE-Agent achieves the highest Resolved Rate of 70.08% (370/528), substantially outperforming both AutoCodeRover and TraeAgent. At the same time, it maintains a high Patch Apply Rate of 95.83% (506/528). This indicates that mini-SWE-Agent combines strong problem-solving capability with stable patch generation, making it the most effective agent on the RACE-bench.

Following the Reasoning Assessment workflow described in Section [2.3](#S2.SS3), we first employ an independent summarizer LLM to normalize the intermediate reasoning process (i.e., Concept Explanations, Goal Expectation, File Localization, Specific Tasks, and Task Steps) of each instance.
If an apply-success patch is generated, we further refine the reasoning chain by applying the same back-inference procedure used to construct the Reasoning Ground Truth (Section [2.2.3](#S2.SS2.SSS3)), deriving intermediate reasoning directly from the generated patch.
We then compare each agent’s reasoning chain against the benchmark-provided Reasoning Ground Truth using the evaluation metrics introduced in Section [2.4](#S2.SS4).
Since modules such as File Localization and Specific Tasks can be reconstructed and supplemented from the generated patch, apply-fail instances prevent structural reconstruction. To ensure fair comparison, we additionally report the failure subset that excludes apply-fail instances.
Table [3](#S4.T3) reports the mean reasoning scores for four groups: All, Success (apply-success and test-success), Failure (apply-fail or test-fail), and Failure (excluding apply-fail).
Our key findings are summarized as follows:
Finding 1. Implicit Conceptual Understanding. Across the intermediate reasoning process of 528 instances, explicit concept explanations are nearly absent. AutoCodeRover provides them in 0.18% (1/528) of cases, TraeAgent in 1.70% (9/528), and mini-SWE-Agent in 0.94% (5/528). In most instances, agents proceed directly to file exploration or code edits without clarifying domain concepts. This pattern reflects an execution-first reasoning style that is shaped by agent prompting. However, all agents achieve consistently high Score@Goal values (9.2–9.6). To reach such high goal expectation scores, the agents must correctly interpret the domain-specific concepts described in the issue. This indicates that the models understand these concepts internally, even though they rarely explain them explicitly in their intermediate reasoning processes.
Finding 2. Strong Intent Understanding but Waterfall Degradation in Execution.
Intent comprehension is not the primary difficulty for repository-level code agents. As shown in Table [3](#S4.T3), all three agents demonstrate a strong high-level understanding of feature requests. The Score@Goal metric remains consistently high across agents, with average scores exceeding 9.2 out of 10 (e.g., 9.506 for TraeAgent and 9.344 for AutoCodeRover). Even in failed instances, these goal comprehension scores show a negligible decline. These results indicate that current agents can accurately capture high-level intent from natural language issues.
However, the main challenge arises when translating high-level intent into concrete implementation plans. We observe a clear waterfall-style degradation in performance as the reasoning chain progresses from file localization to specific tasks and finally to task steps. For example, although mini-SWE-Agent achieves a strong Recall@Files of 0.890, its coverage decreases to 0.751 for Recall@Tasks and further drops to 0.445 for Recall@Steps. At the same time, OverPred@Steps increases across all models, reaching 0.652 for TraeAgent and 0.584 for mini-SWE-Agent in the overall setting. These findings suggest that while identifying high-level intent is comparatively easy for code agents, accurately planning how to execute it step by step remains a significant challenge.
Finding 3. Precision and Redundancy Control Drive Success.
By comparing the Success and Failure (excluding apply-fail) subgroups in Table [3](#S4.T3), we observe that successful instances are characterized by high recall and low over-prediction. Specifically, they maintain Recall@Files above 0.950, Recall@Tasks above 0.840 and Recall@Steps above 0.510. In contrast, apply-success but test-fail cases show a decline in recall and an increase in over-prediction. TraeAgent clearly illustrates this pattern with lower Recall@Files (0.955 0.640) and higher OverPred@Files (0.321 0.568). Similar trends are observed across other reasoning modules (i.e., tasks and steps).
Averaged across all agents and reasoning modules, apply-success but test-fail instances are characterized by a 35.7% overall decrease in recall and a 94.1% increase in over-prediction compared to successful instances. These results suggest that functional failure is associated with imprecise change localization and an excessively broad modification scope.
Finding 4. Distinct Reasoning Styles Across Agents. The three agents exhibit clearly different reasoning behaviors, reflecting trade-offs between exploration and exploitation. AutoCodeRover represents a conservative style. It records low over-prediction scores across all reasoning modules. For example, its overall OverPred@Files is only 0.067 and OverPred@Tasks is 0.255. This strict restriction on redundant edits directly explains its high Patch Apply Rate of 96.21% observed in RQ1. The agent only modifies code entities when it is highly confident. However, this conservative exploitation also limits recall, leading to missed necessary changes. In contrast, TraeAgent adopts an exploratory strategy. Although it achieves a high Resolved Rate (52.65%), its over-prediction is substantially higher, especially at the file and step levels (OverPred@Files = 0.420, OverPred@Steps = 0.652). This pattern indicates that TraeAgent tends to explore more files and task steps, but with weaker control over modification boundaries, which may contribute to its lower Patch Apply Rate (78.98%). mini-SWE-Agent demonstrates a more effective balance between coverage and precision. It achieves the highest recall across reasoning modules (Recall@Files = 0.890 and Recall@Tasks = 0.751), while keeping over-prediction relatively low (OverPred@Files = 0.049 and OverPred@Tasks = 0.208). This combination of broad coverage and controlled redundancy contributes to the higher Resolved Rate (70.08%) observed in RQ1.
Finding 5. Implementation Diversity and Equifinality in Successful Patches. Even in successful cases where patches pass all tests, recall at the task and step levels does not reach 1.0. For example, within the Success subgroup, Recall@Tasks ranges from 0.841 to 0.858 across agents, and Recall@Steps ranges from 0.515 to 0.565. These results indicate that agents can achieve functional correctness without strictly following the Reasoning Ground Truth. This observation reflects a fundamental characteristic of software development: a functional objective can be implemented through multiple implementation paths. An agent can deviate from the ground truth but still produce a successful patch. Consequently, the recall metrics in RACE-bench evaluate alignment with human reasoning rather than strict matching to a single prescribed solution.
Finding 6. Step-Level Reasoning Alignment is the Strongest Predictor of Patch Success. Across all evaluated agents, the most significant divergence between Success and Failure (excluding apply-fail) subgroups occurs at the step decomposition level. Specifically, regardless of the final test result, Score@Goal remains stable across cases. In contrast, Recall@Files decreases by an average of 23.3% in apply-success but test-fail cases, and Recall@Tasks decreases by 34.7%. The largest drop occurs at the step level, where Recall@Steps declines by 49.0%. These results indicate that accurate step decomposition plays an important role in determining whether a repository-level feature can be successfully implemented.

To understand how the underlying LLM influences agent behavior, we evaluate three agents under different base LLMs.
Beyond reporting overall performance, we analyze the intermediate reasoning process of Failure (excluding apply-fail) cases to investigate why patches fail the tests.
The results are shown in Table [4](#S4.T4).
Finding 1. Base LLM Impact Depends on Agent Architecture. The choice of the base LLM leads to performance variations. For example, the Resolved Rate of AutoCodeRover varies from 0.190 (GLM) to 0.370 (GPT), while TraeAgent shows a wider variation from 0.520 (DeepSeek) to 0.770 (GPT). In contrast, mini-SWE-Agent remains relatively stable across models, with Resolved Rate ranging from 0.710 to 0.790. These results indicate that no single base LLM consistently performs best across agents, and the agent framework determines the operational boundary of the base LLM. In conservative agents such as AutoCodeRover, strict AST-guided navigation restricts exploration and causes different LLMs to exhibit similar reasoning behaviors. For instance, AutoCodeRover shows consistent reasoning patterns across models, with Recall@Steps ranging only from 0.280 to 0.362 and OverPred@Steps remaining high (0.661–0.694). This indicates that the reasoning trajectory is largely determined by the agent framework rather than the underlying model. In contrast, a more balanced architecture can reduce the system’s sensitivity to the base LLM. mini-SWE-Agent shows relatively stable performance across models, achieving Resolved Rate of 0.710 (DeepSeek), 0.790 (GLM), and 0.710 (GPT) while maintaining high Patch Apply Rate (0.970–0.990). This finding suggests that well-designed agent architectures can mitigate differences in model capability and achieve more robust cross-model performance.
Finding 2. Stronger Base LLMs Improve Different Agents Through Different Mechanisms. The benefits of stronger base LLMs vary depending on the agent design. For conservative agents such as AutoCodeRover, improvements from GPT primarily appear on the recall side, increasing coverage of relevant tasks and steps. For example, Recall@Steps increases from 0.280 to 0.362, while the reduction in over-prediction remains relatively small (OverPred@Steps decreases from 0.694 to 0.661). This pattern indicates that stronger models help AutoCodeRover identify more required modifications. In contrast, exploratory agents benefit through improved redundancy control. For TraeAgent, switching to GPT reduces unnecessary exploration. OverPred@Files drops from 0.415 (DeepSeek) to 0.000, while OverPred@Tasks decreases from 0.361 to 0.156. These results suggest that stronger base LLMs improve exploratory agents mainly by reducing redundant modifications.
Finding 3. Base LLMs Exhibit Distinct Reasoning Styles. Different base LLMs consistently exhibit distinct behavioral patterns. GPT tends to produce more precise modifications, resulting in lower over-prediction across multiple settings. This effect is most evident in TraeAgent, where OverPred@Files decreases from 0.415 to 0.000 with GPT, while OverPred@Tasks declines from 0.361 to 0.156. These results suggest that GPT is more effective at restricting the modification scope and avoiding unnecessary exploration. In contrast, GLM consistently achieves the lowest Score@Goal across all agents, averaging 7.9% lower than the other base LLMs. This pattern indicates weaker alignment with the overall task objective, making GLM more prone to deviating from the intended feature goal. Overall, these results suggest that different base LLMs introduce distinct reasoning styles into repository-level code agents.

5.1. Efficiency and Cost Analysis
Beyond effectiveness, the practical utility of code agents also depends on their execution efficiency and token consumption. To analyze this aspect, we measure the time and token cost of each agent under three base model settings on RACE-bench Lite. The results are summarized in Table [5](#S5.T5).
In addition, we examine the interaction cost of interactive agents (i.e., TraeAgent and mini-SWE-Agent) that iteratively interact with the environment through multi-turn reasoning–action loops. Specifically, we compute the number of conversation messages, which are defined as non-code textual exchanges between the agent and the base LLM. The results are presented in Figure [3](#S5.F3).
In terms of execution time, DeepSeek is consistently the slowest base model. Across the three agents, the average execution time is 729.0 seconds per instance, which is 56.9% higher than the overall mean runtime of 464.5 seconds. In contrast, GLM is the most efficient model, with an average runtime of 291.4 seconds, representing a 37.3% reduction relative to the overall mean runtime. This difference is likely attributable to the slower API response latency of DeepSeek compared with the relatively faster responses of GLM.
Regarding token consumption, interactive agents incur substantially higher token costs than AutoCodeRover. For example, under DeepSeek, TraeAgent consumes 3,485.9K tokens per instance and mini-SWE-Agent consumes 1,897.6K tokens, which are 18.7 and 10.2 higher than AutoCodeRover (186.0K), respectively.
Interestingly, when the base model is GPT, the token consumption of interactive agents decreases dramatically. Compared with DeepSeek, token usage drops by 78.4% for TraeAgent (from 3,485.9K to 751.1K) and by 86.6% for mini-SWE-Agent (from 1,897.6K to 253.9K). This trend is consistent with Figure [3](#S5.F3), where the average number of conversational messages decreases substantially under GPT.
A closer inspection of the interaction logs suggests that this reduction is driven by differences in response behavior. When interacting with these agents, GPT more frequently produces executable bash commands directly, rather than first explaining its reasoning steps in natural language. This behavior shortens the reasoning–action loop, reduces conversational overhead, and consequently leads to substantially lower token consumption.
5.2. Threats to Validity
5.2.1. Internal Validity
A potential threat arises from the use of the LLM-as-Judge in reasoning assessment, which may introduce evaluator bias in semantic matching. To mitigate this risk, we adopt a hybrid evaluation strategy. For tasks involving structured outputs (e.g., Specific Tasks), we combine deterministic matching rules with semantic similarity checks, thereby reducing reliance on purely LLM-based judgments.
Another threat comes from the construction of the Reasoning Ground Truth, as several reasoning modules are generated with LLM assistance and may therefore contain inaccuracies or omissions. To mitigate this threat, we complement LLM-assisted construction with patch-driven analysis and further perform an LLM-assisted audit over all instances, followed by targeted human validation (Section [2.2.4](#S2.SS2.SSS4)) to verify the semantic correctness and consistency of the resulting Reasoning Ground Truth.
A further threat concerns potential randomness in agent evaluation. In our experiments, we report results from a single run for each instance, which may introduce variance. However, this design follows the first-attempt evaluation setting commonly adopted in existing agent benchmarks (Li et al., [2025](#bib.bib17); Jimenez et al., [2023](#bib.bib16)), reflecting realistic deployment scenarios in which agents are expected to solve tasks without repeated retries.
In addition, we set the generation temperature to 0 to ensure more stable and reproducible results.
5.2.2. External Validity
A threat to external validity is that our experiments cover only a limited set of agents and base models. To mitigate this threat, we select agents with diverse design philosophies and base models from different providers. Future work may extend the evaluation to additional agents and models as the ecosystem continues to evolve. Another threat is that our benchmark evaluates only the explicit intermediate reasoning exposed by agents, whereas agents may also rely on implicit reasoning that is not directly observable. To mitigate this limitation, we refine intermediate reasoning by analyzing the final code patches generated by the agents. This approach allows us to approximate the underlying reasoning process even when parts of it are not explicitly reported. The final threat arises from differences in how agents expose their intermediate reasoning processes, which may affect the fairness of cross-agent comparison. To mitigate this threat, we map the intermediate reasoning process into unified reasoning modules, making the evaluation more comparable across different agent frameworks.
6. Related Work
6.1. Code Generation Benchmarks
Code generation is a core task in software engineering, and a wide range of benchmarks have been proposed to evaluate the code generation capabilities of LLMs.
Early benchmarks focus on single-function synthesis from natural language descriptions and unit tests, such as HumanEval (Chen, [2021](#bib.bib13)) and MBPP (Austin et al., [2021](#bib.bib14)). These benchmarks emphasize algorithmic correctness within a constrained setting.
Subsequent benchmarks extend this paradigm by introducing more complex problems and richer test suites, including APPS (Hendrycks et al., [2021](#bib.bib34)), EvalPlus (Liu et al., [2023a](#bib.bib35)), and CoderEval (Yu et al., [2024](#bib.bib36)).
Other benchmarks, including ClassEval (Du et al., [2023](#bib.bib37)), BigCodeBench (Zhuo et al., [2024](#bib.bib38)), and FullStackBench (Cheng et al., [2024](#bib.bib39)), further expand the evaluation scope to class-level abstractions, large-scale codebases, and full-stack development scenarios.
Despite these advances, most benchmarks still ignore repository context and evaluate code generation in isolation, limiting their ability to reflect realistic scenarios where models must reason over large repositories and generate multi-file changes.
To address this gap, recent work has proposed repository-level code generation benchmarks, such as FEATBench (Chen et al., [2025](#bib.bib28)), DevEval (Li et al., [2024b](#bib.bib40)), EvoCodeBench (Li et al., [2024a](#bib.bib41)), CodevBench (Pan et al., [2024](#bib.bib42)), and ExecRepoBench (Yang et al., [2024a](#bib.bib43)).
Another line of work constructs benchmarks directly from real-world issue–PR pairs.
SWE-bench (Jimenez et al., [2023](#bib.bib16)) extracts bug-fixing tasks from GitHub issues and corresponding patches, enabling end-to-end evaluation of automated software maintenance.
Similarly, FEA-bench (Li et al., [2025](#bib.bib17)) focuses on feature addition tasks derived from real repositories, capturing the challenges of introducing new functionality while preserving existing behavior.
However, these benchmarks primarily focus on whether the final generated patch passes the test suite, while largely ignoring the quality of the intermediate reasoning process.
In contrast, our benchmark augments repository-level tasks with structured reasoning ground truth, enabling systematic evaluation of intermediate reasoning quality and facilitating fine-grained error diagnosis.
6.2. Code Agents
Automatically resolving software issues is a critical problem in software engineering and has attracted increasing research interest in recent years. Recent work extends LLMs from single-turn code generation to repository-level code agents, which operate as autonomous problem solvers over real-world codebases.
Several representative systems have been proposed.
OpenHands (Wang et al., [2024](#bib.bib20)) and SWE-agent (Yang et al., [2024b](#bib.bib22)) model software development as an interactive agent loop with tool use and execution feedback.
Subsequent work investigates more structured forms of guidance for code agents.
AutoCodeRover (Zhang et al., [2024](#bib.bib19)) emphasizes precise navigation by leveraging program structure, such as AST and API dependencies, to guide localized code modifications.
SpecRover (Ruan et al., [2024](#bib.bib44)) further incorporates specification-aware reasoning to steer patch generation toward intended behavioral changes.
In parallel, alternative designs question the necessity of complex agent loops.
Agentless (Xia et al., [2025](#bib.bib45)) shows that competitive performance can be achieved without iterative interaction, instead relying on a carefully designed multi-stage pipeline for localization and repair.
TraeAgent (Gao et al., [2025](#bib.bib18)) explores test-time scaling and iterative refinement strategies to improve robustness in solving software engineering tasks.
Finally, industrial systems such as MarsCode (Liu et al., [2024b](#bib.bib46)) and Claude Code (Anthropic, [2026](#bib.bib47)) further demonstrate practical adoption.
Most existing code agents are evaluated using benchmarks such as SWE-bench (Jimenez et al., [2023](#bib.bib16)), which assess success solely based on whether the final generated patch passes the associated test suite.
Our RACE-bench offers a complementary perspective on evaluating code agents by assessing their capabilities through the quality of the intermediate reasoning process.
7. Conclusion
In this paper, we present RACE-bench, a reasoning-augmented benchmark for evaluating repository-level code agents on real-world feature addition tasks. RACE-bench contains 528 instances from 12 open-source GitHub repositories, each paired with executable verification and structured reasoning ground truth, enabling evaluation at both the patch level and the intermediate reasoning level. Our experiments on three representative agents show substantial differences in end-to-end capability, with Resolved Rate ranging from 29% to 70%. More importantly, the results demonstrate that evaluating only final test outcomes is insufficient for understanding agent behavior. Reasoning-level analysis reveals a clear waterfall-style degradation: agents generally perform well at high-level goal understanding, but become much weaker when translating goals into files, tasks, and especially concrete implementation steps. Step-level reasoning is the most critical bottleneck, and test-fail instances are often associated with lower recall (a 35.7% decrease) and excessive over-prediction (a 94.1% increase). We further find that the impact of the base LLM is not uniform across agents. Instead, it is strongly shaped by the agent architecture, suggesting that model capability and agent design should be studied jointly rather than in isolation. Overall, RACE-bench provides a practical foundation for diagnosing, comparing, and improving repository-level code agents beyond black-box patch success evaluation.
