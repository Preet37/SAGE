## Core Definitions

**Agent streaming**: In LangGraph/LangSmith, “streaming” is the ability to emit partial outputs during a graph run—ranging from token-by-token model output to step-by-step state changes—so the user can see progress before the run completes. LangGraph exposes this via `graph.stream(...)` / `graph.astream(...)` with `stream_mode` options such as `values`, `updates`, `messages`, `debug`, `custom`, and `checkpoints`. (LangGraph runtime + streaming modes: https://blog.langchain.com/building-langgraph/ ; Reliable streaming announcement: https://changelog.langchain.com/announcements/reliable-streaming-and-efficient-state-management-in-langgraph)

**Token streaming**: Token streaming is streaming the model’s generated text incrementally as it is produced. In LangSmith run streaming, this corresponds to `stream_mode="messages-tuple"` where each chunk contains `(message_chunk, metadata)`; UIs typically print `message_chunk["content"]` while using metadata (e.g., node name) for routing/filtering. (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)

**Step streaming (state update streaming)**: Step streaming is emitting information at graph step boundaries (and sometimes multiple times within a step) to show intermediate progress. In LangSmith run streaming, `stream_mode="updates"` streams state *updates* after each step (and can stream multiple updates per step if multiple nodes update state). `stream_mode="values"` streams the *full state* after each super-step. (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming; Reliable streaming announcement: https://changelog.langchain.com/announcements/reliable-streaming-and-efficient-state-managem.card.md)

**Intermediate state inspection**: Intermediate state inspection is the practice of examining the graph’s evolving state (e.g., message history, tool outputs, routing decisions) at checkpoints/snapshots to debug behavior. When compiled with a checkpointer, LangGraph saves a `StateSnapshot` at each super-step, which can be retrieved later to see `values`, `metadata.step`, and what node(s) are next. (LangGraph discussion w/ snapshots: https://github.com/langchain-ai/langgraph/discussions/2028; JS checkpointer reference: https://reference.langchain.com/javascript/langchain-langgraph/index/CompiledStateGraph/checkpointer)

**Graph debugging**: Graph debugging is diagnosing incorrect or unexpected agent behavior by inspecting node-level execution, state transitions, and streamed debug information. LangGraph/LangSmith support a `debug` stream mode that emits “as much info as possible,” including node name and full state. (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming; Reliable streaming announcement: https://changelog.langchain.com/announcements/reliable-streaming-and-efficient-state-management-in-langgraph)

**Execution tracing**: Tracing is recording a run as a sequence of steps (“runs”) that show how the application went from input to output; each step is represented as a run in LangSmith’s model. LangSmith traces let you visualize execution steps for debugging, evaluation, and monitoring. (LangSmith Observability docs: https://docs.langchain.com/oss/python/langgraph/observability)

**Observability (production monitoring for agents)**: Observability is the ability to explain and measure production behavior of an agent system using traces, metadata/tags, and monitoring/evaluation. LangSmith positions tracing as enabling local debugging, performance evaluation, and production monitoring (dashboards). (LangSmith Observability docs: https://docs.langchain.com/oss/python/langgraph/observability; LangSmith Evaluation overview: https://docs.langchain.com/langsmith/evaluation)

## How It Works

### A. LangGraph streaming at the graph level (`graph.stream` / `graph.astream`)

1. **Compile a graph** (optionally with a checkpointer for persistence/time-travel).
   - Without a checkpointer, each invocation is effectively “fresh” state unless you manage state externally.
   - With a checkpointer, LangGraph saves a checkpoint (snapshot) at each **super-step** boundary.  
   (JS checkpointer reference: https://reference.langchain.com/javascript/langchain-langgraph/index/CompiledStateGraph/checkpointer)

2. **Invoke with streaming**:
   - Use `graph.stream(inputs, config, stream_mode=...)` (sync) or `graph.astream(...)` (async).
   - Choose `stream_mode` based on what you want to emit:
     - `values`: full state after each super-step
     - `updates`: only changed keys (may emit multiple times per step)
     - `messages`: streaming message chunks + metadata from LLM calls
     - `custom`: your own events from inside nodes
     - `checkpoints`: checkpoint events (requires checkpointer)
     - `debug`: node name + full state (“as much info as possible”)  
   (Reliable streaming announcement: https://changelog.langchain.com/announcements/reliable-streaming-and-efficient-state-management-in-langgraph; LangGraph design blog: https://blog.langchain.com/building-langgraph/)

3. **Consume stream parts**:
   - For `updates`, treat each part as a partial patch to state.
   - For `values`, treat each part as a complete snapshot.
   - For `messages`, treat each part as incremental model output; use metadata to route/filter by node.

4. **(Optional) Stream subgraphs**:
   - Enable subgraph streaming to see both parent and subgraph outputs; streamed parts include a namespace (`ns`) to distinguish root vs subgraph.  
   (Reliable streaming announcement: https://changelog.langchain.com/announcements/reliable-streaming-and-efficient-state-management-in-langgraph; LangSmith streaming API mentions `stream_subgraphs=True`: https://docs.langchain.com/langsmith/streaming)

### B. LangSmith “run streaming” (API-level streaming tied to threads/runs)

This is the “production wiring” pattern when using LangSmith deployments / assistants.

1. **Create a client**:
   ```python
   client = get_client(url=<DEPLOYMENT_URL>, api_key=<API_KEY>)
   ```
   (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)

2. **Create a thread (stateful)**:
   ```python
   thread = await client.threads.create()
   thread_id = thread["thread_id"]
   ```
   Threads group checkpoints/state across runs. (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)

3. **Stream a run**:
   ```python
   async for chunk in client.runs.stream(
       thread_id,
       assistant_id,
       input=inputs,
       stream_mode="updates",
   ):
       print(chunk.data)
   ```
   (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)

4. **Stateless run option**:
   - Pass `None` instead of `thread_id` to avoid persisting outputs in the checkpointer DB:
     ```python
     client.runs.stream(None, assistant_id, input=inputs, stream_mode="updates")
     ```
   (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)

5. **Token streaming via `messages-tuple`**:
   - Shape: `chunk.data == (message_chunk, metadata)`
   - Typical UI behavior: print `message_chunk["content"]`; use metadata (e.g., node info) for filtering/routing.  
   (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)

6. **Multi-mode streaming**:
   - Request multiple modes:
     ```python
     stream_mode=["updates", "custom"]
     ```
   - Output becomes tuples `(mode, chunk)`.  
   (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)

7. **Join an existing run’s stream**:
   - `client.runs.join_stream(thread_id, run_id)`
   - Important: outputs are **not buffered**; you won’t receive earlier chunks that already streamed.  
   (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)

### C. Debugging via checkpoints / snapshots (intermediate state inspection)

1. **Compile with a checkpointer**:
   - Example shown in community guidance:
     ```python
     graph = graph_builder.compile(checkpointer=MemorySaver())
     ```
   (LangGraph issue #1568 discussion: https://github.com/langchain-ai/langgraph/issues/1568; LangGraph discussion: https://github.com/langchain-ai/langgraph/discussions/2028)

2. **Provide a `thread_id` in config** so state persists across turns:
   ```python
   thread_config = {"configurable": {"thread_id": "1"}}
   for event in graph.stream(input_state, thread_config):
       ...
   ```
   Without this, the graph “restarts after each run,” so message history appears overwritten rather than appended. (LangGraph issue #1568: https://github.com/langchain-ai/langgraph/issues/1568)

3. **Inspect snapshots**:
   - A `StateSnapshot` includes:
     - `values` (full state)
     - `metadata.step` (step number)
     - `next` (empty at `END`)
     - checkpoint identifiers (`checkpoint_id`, `checkpoint_ns`)  
   (LangGraph discussion: https://github.com/langchain-ai/langgraph/discussions/2028)

### D. Enabling observability (LangSmith tracing)

1. **Enable tracing globally**:
   ```bash
   export LANGSMITH_TRACING=true
   export LANGSMITH_API_KEY=<your-api-key>
   ```
   (LangSmith Observability docs: https://docs.langchain.com/oss/python/langgraph/observability)

2. **Trace selectively** with `tracing_context`:
   ```python
   import langsmith as ls

   with ls.tracing_context(enabled=True):
       agent.invoke({"messages": [{"role": "user", "content": "Send a test email"}]})
   ```
   Calls outside the context won’t be traced unless global env vars are set. (LangSmith Observability docs: https://docs.langchain.com/oss/python/langgraph/observability)

3. **Log to a project**:
   - Static: `LANGSMITH_PROJECT=my-agent-project`
   - Dynamic:
     ```python
     with ls.tracing_context(project_name="email-agent-test", enabled=True):
         agent.invoke(...)
     ```
   (LangSmith Observability docs: https://docs.langchain.com/oss/python/langgraph/observability)

## Teaching Approaches

### Intuitive (no math)
- **Streaming** answers: “What can I show the user *while* the agent is thinking?”  
  - Tokens: show partial text.
  - Updates: show progress (“retrieving…”, “calling tool…”, “drafting answer…”).
  - Values/debug: show internal state for developers.
- **Debugging** answers: “Where did the agent go wrong?”  
  Use step updates + checkpoints to pinpoint the first bad state transition.
- **Observability** answers: “Can I explain what happened in production after the fact?”  
  Traces + metadata let you reconstruct the run.

### Technical (with runtime mechanics)
- LangGraph executes in **super-steps** (BSP/Pregel-inspired): nodes run (potentially in parallel), then updates are applied deterministically; streaming can emit at step boundaries (`values`/`updates`) and during model calls (`messages`). (LangGraph design blog: https://blog.langchain.com/building-langgraph/)
- Checkpointing persists state at each super-step; threads group checkpoints across runs; `thread_id` is the key for persistence/resume. (JS checkpointer reference: https://reference.langchain.com/javascript/langchain-langgraph/index/CompiledStateGraph/checkpointer)

### Analogy-based
- **Streaming modes are “camera angles”** on the same execution:
  - `messages` = microphone on the model’s mouth (token-by-token).
  - `updates` = diff view in Git (only what changed).
  - `values` = full screenshot after each scene.
  - `debug` = director’s commentary (node names + full state).
- **Tracing is a flight recorder**: you don’t need it to fly, but you need it to investigate incidents and improve reliability. (LangSmith Observability docs: https://docs.langchain.com/oss/python/langgraph/observability)

## Common Misconceptions

1. **“If I annotate `messages` with `add_messages`, it will always append across user turns automatically.”**  
   - Why wrong: Without a **checkpointer + thread_id**, each run starts from scratch, so you only see the current turn’s messages. This shows up as “replaced rather than appended.”  
   - Correct model: `add_messages` defines how updates merge *within a run’s state evolution*, but persistence across turns requires checkpointing and a stable `thread_id`.  
   - Source: LangGraph issue #1568 explains “Insert a checkpoint… Otherwise, your graph will restart after each run.” (https://github.com/langchain-ai/langgraph/issues/1568)

2. **“`values` and `updates` are basically the same; they both stream state.”**  
   - Why wrong: `values` streams the **full state snapshot** after each super-step; `updates` streams **only changed keys**, and can emit multiple updates per step if multiple nodes update state.  
   - Correct model: choose `updates` for efficient progress streaming; choose `values` when you need complete snapshots for debugging/UI state reconstruction.  
   - Source: LangSmith streaming modes definitions (https://docs.langchain.com/langsmith/streaming) and release notes (https://changelog.langchain.com/announcements/reliable-streaming-and-efficient-state-management-in-langgraph)

3. **“Token streaming is always raw model tokens.”**  
   - Why wrong: Some implementations stream “word-like chunks” rather than raw tokens; e.g., the FastAPI WebSocket scaffold explicitly notes tokens are converted to words for display.  
   - Correct model: treat token streaming as “incremental text chunks”; don’t assume chunk boundaries match tokenizer tokens.  
   - Source: LangGraph_Streaming repo note: “not raw token streaming; tokens are converted to words.” (https://github.com/sheikhhanif/LangGraph_Streaming)

4. **“If I join a run stream late, I’ll still get everything that already streamed.”**  
   - Why wrong: `client.runs.join_stream(...)` does **not buffer** earlier output; you only receive events from the point you join.  
   - Correct model: if you need replay, store events yourself or use thread stream resumability (`Last-Event-ID`) where applicable.  
   - Source: LangSmith streaming API: “outputs not buffered (miss earlier output).” (https://docs.langchain.com/langsmith/streaming)

5. **“Tracing is only for debugging locally; it’s too heavy for production.”**  
   - Why wrong: LangSmith frames tracing as enabling not just debugging but also evaluation and monitoring (dashboards) in production. Also, you can trace selectively via `tracing_context` and route traces to projects.  
   - Correct model: use sampling/selection (trace selectively) and metadata/projects to make production tracing manageable.  
   - Source: LangSmith Observability docs (https://docs.langchain.com/oss/python/langgraph/observability)

## Worked Examples

### Example 1 — Python: Persisted conversation + step streaming + intermediate inspection

This example demonstrates the *specific fix* for “messages not appending” and gives a streaming loop you can adapt.

```python
from typing import Annotated
from typing_extensions import TypedDict

from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.checkpoint.memory import MemorySaver

# If using a real model, you'd import it (e.g., ChatOpenAI) as in issue #1568.
# Here we keep the node simple and deterministic.
class State(TypedDict):
    messages: Annotated[list, add_messages]

def mock_llm(state: State):
    # Append an AI message each turn
    last_user = state["messages"][-1]
    return {"messages": [{"role": "ai", "content": f"echo: {last_user.content if hasattr(last_user,'content') else last_user}"}]}

builder = StateGraph(State)
builder.add_node("chatbot", mock_llm)
builder.add_edge(START, "chatbot")
builder.add_edge("chatbot", END)

# Key: compile with a checkpointer
graph = builder.compile(checkpointer=MemorySaver())

# Key: stable thread_id in configurable config
thread_config = {"configurable": {"thread_id": "thread-1"}}

# Stream state updates
for event in graph.stream({"messages": ("user", "hi")}, thread_config):
    # event is a dict of updates by node in many examples; print what you need
    print(event)

for event in graph.stream({"messages": ("user", "and then?")}, thread_config):
    print(event)
```

**What to point out while tutoring**
- The *reason* history persists across turns is the combination of:
  1) `compile(checkpointer=MemorySaver())`
  2) `{"configurable": {"thread_id": ...}}` passed on each call  
  (LangGraph issue #1568: https://github.com/langchain-ai/langgraph/issues/1568)
- If a student wants richer debugging, switch to `stream_mode="values"` or `stream_mode="debug"` (graph-level) or use LangSmith run streaming modes. (Streaming modes: https://docs.langchain.com/langsmith/streaming; https://changelog.langchain.com/announcements/reliable-streaming-and-efficient-state-management-in-langgraph)

### Example 2 — LangSmith run streaming: token streaming + step updates (multi-mode)

```python
# Pseudocode shape from LangSmith docs (async)
client = get_client(url=DEPLOYMENT_URL, api_key=API_KEY)
thread = await client.threads.create()
thread_id = thread["thread_id"]

async for mode, chunk in client.runs.stream(
    thread_id,
    assistant_id,
    input=inputs,
    stream_mode=["messages-tuple", "updates"],
):
    if mode == "messages-tuple":
        message_chunk, metadata = chunk.data
        # UI: print incremental content
        print(message_chunk["content"], end="", flush=True)
    elif mode == "updates":
        # Dev/progress: show state diffs
        print("\nSTATE UPDATE:", chunk.data)
```

**Key tutoring notes**
- Multi-mode returns `(mode, chunk)` tuples. (LangSmith streaming API: https://docs.langchain.com/langsmith/streaming)
- `messages-tuple` chunk shape is `(message_chunk, metadata)`. (Same source)

## Comparisons & Trade-offs

| Goal | Best mechanism | Why | Trade-offs / gotchas | Source |
|---|---|---|---|---|
| Chat UI “typing” effect | `messages` / `messages-tuple` | Token/chunk streaming with metadata for routing | Chunk boundaries may not be raw tokens; may need node filtering | LangSmith streaming API: https://docs.langchain.com/langsmith/streaming ; LangGraph_Streaming repo: https://github.com/sheikhhanif/LangGraph_Streaming |
| Progress UI for long runs | `updates` | Streams diffs; can show tool calls, routing, partial results | Multiple updates can arrive per step; UI must merge patches | LangSmith streaming API; Reliable streaming announcement |
| Developer debugging | `debug` or `values` + checkpoints | Full state + node info; time-travel via snapshots | More data volume; be careful with sensitive data | LangSmith streaming API; JS checkpointer reference |
| Persisted memory across turns | Checkpointer + `thread_id` | Saves checkpoints per super-step in a thread | Without `thread_id`, no persistence; confusion with `add_messages` | LangGraph issue #1568; checkpointer reference |
| Post-hoc production explainability | LangSmith tracing | Visualize runs/steps; add metadata/tags/projects | Must manage sensitive data; consider selective tracing | LangSmith Observability docs |

## Socratic Question Bank

1. **If your UI needs to show “retrieving docs…” before the model starts answering, which stream mode would you choose and why?**  
   *Good answer:* `updates` (or `custom`) because it can emit step-level progress independent of token output.

2. **What’s the practical difference between streaming `values` vs `updates` when building a frontend state store?**  
   *Good answer:* `values` replaces the whole state each time; `updates` requires patch/merge logic and may emit multiple patches per step.

3. **A student says: “My messages aren’t accumulating even with `add_messages`.” What’s the first thing you check?**  
   *Good answer:* whether a checkpointer is configured and a stable `thread_id` is passed each run (issue #1568).

4. **If you join a run stream late, what do you expect to miss, and how would you design around it?**  
   *Good answer:* earlier streamed output isn’t buffered; design with resumable thread streams (Last-Event-ID) or store events yourself.

5. **How would you filter token streaming so only the final generation node streams to the user?**  
   *Good answer:* use event metadata/tags identifying the node; filter on that (LangGraph.js issue #320 points to metadata-based filtering).

6. **What’s the difference between “debugging with streaming” and “debugging with tracing”?**  
   *Good answer:* streaming is real-time visibility; tracing is recorded post-hoc visibility (and can be used for monitoring/evals).

7. **When would you choose a stateless run (`thread_id=None`) even though you lose persistence?**  
   *Good answer:* when you don’t want to persist outputs/state in the checkpointer DB (privacy, cost, ephemeral tasks).

8. **How can you trace only certain invocations in a production service?**  
   *Good answer:* use `langsmith.tracing_context(enabled=True)` selectively and/or project routing.

## Likely Student Questions

**Q: What stream modes exist in LangSmith run streaming, and what do they mean?** → **A:** `values` (full state after each super-step), `updates` (state updates after each step; multiple updates per step possible), `messages-tuple` (token/chunk streaming as `(message_chunk, metadata)`), `debug` (max info incl. node name + full state), `custom` (user-defined streamed data), `events` (all events; used for migrating LCEL apps). (https://docs.langchain.com/langsmith/streaming)

**Q: How do I run a stateless stream that doesn’t persist to the checkpointer DB?** → **A:** Pass `None` as `thread_id` to `client.runs.stream(None, assistant_id, input=..., stream_mode=...)`. (https://docs.langchain.com/langsmith/streaming)

**Q: Why do I see my message history “reset” each user turn in LangGraph?** → **A:** Without a checkpointer, the graph restarts each run; to persist and append messages across turns, compile with a checkpointer (e.g., `MemorySaver()`) and pass a stable `thread_id` in `configurable`. (https://github.com/langchain-ai/langgraph/issues/1568)

**Q: How do I enable LangSmith tracing quickly?** → **A:** Set `LANGSMITH_TRACING=true` and `LANGSMITH_API_KEY=<key>`; traces go to project `default` unless you set `LANGSMITH_PROJECT`. (https://docs.langchain.com/oss/python/langgraph/observability)

**Q: Can I trace only some calls (not everything)?** → **A:** Yes—use `langsmith.tracing_context(enabled=True)` around the invocations you want traced; calls outside won’t be traced unless global env vars are set. (https://docs.langchain.com/oss/python/langgraph/observability)

**Q: If I use `client.runs.join_stream`, will I get earlier output?** → **A:** No—join streams are not buffered; you can miss earlier output if you join late. (https://docs.langchain.com/langsmith/streaming)

**Q: How do I resume/replay a thread stream if my connection drops?** → **A:** Thread streams support resumability via `Last-Event-ID` / `last_event_id="<LAST_EVENT_ID>"`; pass `"-"` to replay from the beginning. (https://docs.langchain.com/langsmith/streaming)

**Q: How do I filter streaming so only the final node’s tokens are shown?** → **A:** Use event metadata identifying the node (and/or tagging) to filter events; LangGraph.js maintainers recommend metadata/tags rather than tracking `currentNode` manually. (https://github.com/langchain-ai/langgraphjs/issues/320)
